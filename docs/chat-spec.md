# ТЗ: модуль чатов (WMess) — реализация с нуля

Реализовать модуль чатов в существующем стеке WMess. Чат-кода в проекте нет — пишем с нуля. Команды, проекты, роли (`TeamRole`) и аутентификация уже есть и переиспользуются.

## Стек и архитектура
- **Бэкенд:** .NET (EF Core + PostgreSQL), JWT.
- **BFF:** WMess.Web (YARP/SpaYarp, cookie-сессия) — клиент ходит через него; SignalR-хаб на API проксируется BFF на `/hubs/*`.
- **Клиент:** React + Vite + TypeScript, zustand, `@microsoft/signalr`.
- **Реалтайм:** SignalR.
- **Авторизация — resource-based:** права в чате выводятся из роли пользователя в команде/проекте (`TeamRole`: Member < Admin < Owner). Логику прав на клиенте не дублировать. Если в проекте есть модуль документов — следовать его паттерну (`*AccessService` + `*Hub`).

## Доменная модель (EF Core)
- **Chat**: Id, Name?, Type (`Team` | `Project` | `Direct`), TeamId?, ProjectId?, MaxNestingLevel (0 = без лимита), CreatedAt.
- **Message**: Id, ChatId, AuthorId, Content (может быть пустым при наличии вложений), ParentMessageId?, CreatedAt, EditedAt?, Transcription?, WaveformData?.
- **Attachment**: Id, MessageId, FileName, StoredName, ContentType, Size.
- **Reaction**: Id, MessageId, UserId, Emoji — уникально по (MessageId, UserId, Emoji).
- **PinnedMessage**: Id, ChatId, MessageId, PinnedBy, PinnedAt — уникально по (ChatId, MessageId).
- **ChatMember**: ChatId, UserId, JoinedAt, LastReadAt?.
- Индексы: Message(ChatId, CreatedAt), Message(ParentMessageId).

## ChatAccessService (единый источник прав)
Используется и контроллерами, и хабом:
- Проектный чат: доступ = доступ к проекту; управление = управление проектом.
- Командный чат: доступ = участник команды; управление = Admin/Owner.
- Иерархия: `CanManage` ⊇ `CanAccess`.
- Видимость чата выводится из членства в команде/проекте, а не из таблицы участников.

## Функциональность
1. **Создание чатов** — привязка к команде ИЛИ проекту; создавать может участник; `Type` выставляется по переданному Team/Project.
2. **Текстовые сообщения** — отправка, редактирование (`EditedAt`), пагинация ленты.
3. **Реакции-эмодзи** — toggle (повторная снимает); событие с явным флагом `added`.
4. **Картинки/видео** — upload (multipart); инлайн-рендер: `image/*` → `<img>`, `video/*` → `<video>`.
5. **Прочие файлы** — те же вложения; в ленте ссылка на скачивание. Не-медиа отдавать как `attachment` (Content-Disposition), лимит размера, проверка доступа к чату при upload **и** download.
6. **Голосовые** — запись на клиенте (MediaRecorder), загрузка как `audio`-вложение.
7. **Диаграмма голосовых (waveform)** — отрисовка волны при проигрывании (напр. wavesurfer.js); пики считать на клиенте или хранить в `WaveformData`.
8. **Расшифровка голосовых** — *сейчас НЕ реализовывать*, но заложить архитектуру: поле `Message.Transcription`, абстракция `ITranscriptionService` и асинхронный хук после загрузки голосового (под будущий Whisper); эндпоинт расшифровки может быть заглушкой.
9. **Закреплённые сообщения** — pin/unpin; право = управление чатом (Admin/Owner); отдельная панель закреплённых; события `MessagePinned`/`MessageUnpinned`.
10. **Ответы** — `ParentMessageId`; в UI превью родителя над полем ввода и переход к нему.
11. **Цитирование** — вставка цитируемого фрагмента в новое сообщение со ссылкой на оригинал (механика отдельная от reply).
12. **Вложенные комментарии (Mattermost)** — треды от корневого сообщения. Правило уровней: на корневое (уровень 1) можно ответить **двумя** способами — тредом (Mattermost) ИЛИ обычным ответом (Telegram); на ответы (уровень 2+) — только Telegram-style (плоский ответ с цитатой, без углубления). При отправке проверять: `ParentMessageId` принадлежит этому чату И глубина ≤ `MaxNestingLevel`.
13. **Созвоны** — аудио/видео-звонки в чате (Jitsi или WebRTC): кнопка старта создаёт комнату/ссылку и кладёт сообщение-приглашение в ленту.
14. **Ссылки на задачи через `#` (на будущее)** — заложить архитектурно: парсинг `#{номер}` в тексте → резолв в карточку задачи (красивое название + урезанное описание) через `ITaskResolver`, рендер карточки в сообщении. Резолвер пока заглушка. Контент проектировать с поддержкой инлайн-сущностей/упоминаний.

## API и реалтайм
- **REST:** создать чат; список чатов; лента сообщений (пагинация); отправить/редактировать сообщение; реакция (toggle); pin/unpin; upload/download вложения.
- **SignalR:** `ReceiveMessage`, `MessageUpdated`, `ReceiveReaction` (с `added`), `MessagePinned`, `MessageUnpinned`, `UserTyping`.
- Мутации (сообщения/реакции/пины) идут через **REST** (сохраняет в БД и рассылает событие через `IHubContext`); хаб — только подписка на группу чата и typing. Каждый REST-эндпоинт и метод хаба проверяет доступ к чату.

## Безопасность / нюансы
- Права проверять на каждый вызов; не допускать IDOR (проверять, что сообщение/вложение принадлежит чату).
- Сообщение допустимо без текста, если есть вложение.
- Уникальные ограничения на реакции и пины; лимит размера файлов; не-медиа — только на скачивание.

## Рекомендуемый порядок реализации
1. Модель + миграции. 2. `ChatAccessService`. 3. REST + SignalR для текста (отправка/лента/редактирование). 4. Реакции, пины, ответы. 5. Вложения (картинки/видео/файлы). 6. Голосовые + waveform. 7. Треды + правило уровней. 8. Созвоны. 9. Заглушки на будущее (расшифровка, `#`-задачи).
